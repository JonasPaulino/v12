Sobre usar Next.js

Next.js é bom para web, mas eu não usaria Next.js puro como PDV desktop local.

Ele pode juntar front e back, sim, mas para instalar em cliente você ainda vai cair em problemas como:

porta local;

Para PDV, eu iria de:

Melhor opção: Electron + React + Node

Você já trabalha com React e Node. Então faria sentido usar:

Electron para desktop;
React no front;
Node no processo principal;
SQLite local;
integração com ACBrMonitorPLUS ou ACBrLib;
empacotamento com instalador .exe.

Isso permite usar uma linguagem principal: JavaScript/TypeScript.

O Electron não protege 100% o código; arquivos ASAR podem ser extraídos, e isso é uma limitação conhecida. Dá para dificultar com empacotamento, obfuscação, assinatura, V8 bytecode e validação de licença, mas não dá para considerar “inviolável”.

Sobre ACBr

Você não precisa voltar para Lazarus/Pascal.

Pode usar:

Opção 1 — ACBrMonitorPLUS

Seu sistema local conversa com o ACBr por arquivo TXT, socket ou comandos.

Vantagem:

mais simples;
funciona com qualquer linguagem;
bom para Node/Electron;
menos dependência de compilar DLL.

O próprio ACBr tem materiais sobre integração com qualquer linguagem e contingência NFC-e offline.

Opção 2 — ACBrLib

Mais profissional, usando DLL.

Vantagem:

integração mais direta;
menos dependente de um monitor aberto.

Desvantagem:

mais trabalho com DLL, configuração e distribuição.

Para começar, eu iria de ACBrMonitorPLUS. Depois, se o V12 crescer, você pode migrar para ACBrLib.

Como funcionaria a contingência

Fluxo ideal:

Cliente compra
↓
PDV tenta emitir NFC-e online
↓
Se SEFAZ/internet falhar:
    emite em contingência offline
    imprime cupom
    grava XML local
    marca como "pendente de envio"
↓
Quando internet voltar:
    transmite para SEFAZ
    atualiza status no V12 online

A contingência offline da NFC-e existe justamente para falha de comunicação com a SEFAZ, com transmissão posterior.

O que mais me anima na sua ideia

Na minha opinião, o maior diferencial é que você não está pensando em criar apenas um emissor de NFC-e. Você está pensando em um ecossistema:

V12 ERP → gestão completa na nuvem.
V12 PDV → operação local, rápida e resiliente.
Sincronização automática → os dois trabalham como um único sistema.

É exatamente esse modelo que muitos ERPs consolidados utilizam. A diferença é que, como você controla tanto o ERP quanto o PDV, pode fazer uma integração muito mais limpa, compartilhando componentes, lógica e APIs entre os dois projetos. Isso reduz bastante o esforço de manutenção e evolução ao longo do tempo.
